多模态稀疏 3D 检测：工程架构深度解析

2026-04-10 · 返回首页

自动驾驶感知系统的核心挑战在于：如何在有限算力下，将多传感器的异构数据高效融合为统一的 3D 世界表示。本文围绕一套量产级多模态 BEV（Bird's Eye View）3D 目标检测系统，从工程架构设计、数据流转逻辑和关键技术决策三个维度，剖析其背后的设计思考。

一、总体架构：从传感器到检测结果的数据流

整个系统遵循一条清晰的主线：

原始传感器数据 → 单模态特征提取 → BEV 空间投影与融合 → 稀疏检测头 → 3D 目标框

但这条主线背后隐藏着精妙的多分支、多尺度、多任务设计。系统的骨架是一个多任务共享 Backbone 框架：视觉主干和 LiDAR 主干是所有下游任务（OD、Occupancy、LidarGod 等）的公共基础设施，通过独立的 FPN Neck 分流到不同任务头。这种设计的工程收益显而易见 —— Backbone 只需前向一次，多任务共享计算。

从数据维度看，系统要处理的输入规模相当可观：12 路相机（前视 8 路、后视 5 路中取 3 路环视、加上裁切视图）的图像，加上一颗 LiDAR 的百万级点云。系统通过三组独立的视觉 Backbone（前视、后视、环视）分别处理不同位置和分辨率的相机图像，再经过各自的 FPN 提取多尺度特征。这种"分组独立处理"的策略背后有一个实际考量：不同位置的相机在安装角度、分辨率和关注区域上差异巨大，共享一个 Backbone 反而会导致特征表达的互相妥协。

二、LiDAR 稀疏处理管线：从点云到 BEV 的五步转化

LiDAR 处理管线是整个系统中最精巧的部分。它解决了一个根本性问题：原始点云是无序的、密度不均匀的，而下游检测需要规整的、多尺度的特征图。

这个转化通过五个串联阶段完成：

2.1 点云到距离图：将无序变为有序

第一步并非直接体素化，而是先将三维点云投影到二维距离图（Range Image）。这一设计的直觉是：LiDAR 的扫描本质上是球面扫描 —— 每个点对应一个方位角和俯仰角，天然具有二维结构。通过球面投影，我们将无序点云变成了一张 160×1120 的伪图像，可以利用成熟的 2D 卷积来提取特征。

这里有一个关键的非均匀设计：垂直方向采用 ROI 模式，地面附近的角度区间（5.2°~14.8°）分配了更密的像素行，高空区域则稀疏采样。这与 LiDAR 硬件的扫描模式精确匹配 —— 在地面和近距离目标密集的区域保留最高分辨率，避免信息损失。

2.2 Range View U-Net：给每个点注入全局上下文

距离图上运行一个轻量 U-Net（4 层下采样、3 层上采样），感受野从单个像素扩展到覆盖数十个扫描线。然后通过一个反向索引操作（Gather），将 U-Net 输出的特征逐点回投到原始点云。

这一步的核心洞见是：直接体素化的点云缺乏跨扫描线的感知能力。一个体素只能看到自身包含的几个点，对相邻空间区域一无所知。而 Range View U-Net 等效于在体素化之前，用一个基于自然扫描结构的轻量网络来丰富每个点的特征，使其具备多尺度表面感知能力。经过这一步，每个点从 6 维原始特征扩展为 14 维增强特征。

这里的 Scatter-Gather 模式值得注意：

  • Scatter（聚合）：多个点投影到同一像素时取均值，形成距离图
  • Gather（分配）：U-Net 处理后，每个点通过记录的索引取回对应像素的特征

多个点可能映射到同一像素（遮挡关系），它们会共享该像素的 U-Net 输出特征。这是一种有意的信息共享 —— 处于同一角度方向的点本就具有空间相关性。

2.3 可微体素化：梯度的生命线

将 14 维增强点云转化为稀疏体素时，系统采用了支持梯度反传的体素化实现。这不是一个微小的实现细节，而是一个架构级的关键决策。

传统体素化使用 CUDA 自定义算子，计算图在此处断裂，梯度无法回传到前面的 U-Net。这意味着 Range View 分支在训练中形同冻结。可微体素化的实现思路是：用原生的确定性索引累加操作替代不透明的 CUDA 算子，让整条管线从稀疏卷积的 Loss 一直反传到 U-Net 参数。

这里还有一个工程细节：体素坐标的线性化索引（将 4D 坐标展平为 1D 整数）必须使用 float64 计算。原因是体素网格的规模（2560×704×54×2 ≈ 1.77 亿）超出了 float32 能精确表示的最大整数（224 ≈ 1677 万）。这种数值精度问题在大规模量产系统中尤为常见，但容易被忽略。

另一个值得关注的设计是确定性累加。标准的 CUDA 原子操作存在竞态，同一输入多次运行可能产生微小的数值差异。系统使用了确定性版本（预排序后串行累加），以约 10-20% 的速度代价换取完全可复现的训练结果 —— 这对实验对比和问题调试至关重要。

2.4 稀疏 3D 卷积：多尺度几何特征提取

增强后的体素特征经过 6 层稀疏 3D 卷积，逐步降采样并提取不同尺度的几何特征。关键在于"稀疏"二字 —— 8800 万个体素格子中实际被占用的不超过 1%，稀疏卷积只在有数据的体素上计算，避免了天文数字般的无效运算。

网络输出三个尺度的稀疏特征：

  • s2 尺度（步幅 2）：保留细粒度几何，覆盖近场
  • s4 尺度（步幅 4）：中等粒度，覆盖中场
  • s8 尺度（步幅 8）：粗粒度但覆盖全范围

2.5 稀疏到密集：BEV 特征图生成

最后一步将稀疏特征转化为密集 BEV 特征图，三个尺度的处理策略各不相同：

  • s2：Z 维取最大值（max pooling）→ 消除高度，保留最显著特征
  • s4：同样 Z 维取最大值
  • s8：Z 维通道拼接（concat）→ 保留高度信息编码

s8 尺度保留高度信息的原因是：远距离目标的水平分辨率已经很低，高度方向的信息对区分车辆和高架等结构尤为重要。

另一个值得注意的细节是非对称裁剪：s2 和 s4 尺度在 X 方向（前进方向）只取前半段，分别覆盖 0~77m 和 0~154m。近场和中场的检测精度要求最高，同时裁剪后特征图尺寸减半，降低后续计算量。只有 s8 尺度覆盖完整的 0~307m 范围。

三、多模态融合：相机与 LiDAR 的协同策略

系统的融合策略并非简单的特征拼接，而是根据不同任务的需求采用了差异化的融合深度。

3.1 BEV 空间投影

相机图像首先通过可变形注意力机制（类 LSS 方法）从前视图（FOV）空间投影到 BEV 空间。这一模块在微调阶段被冻结 —— 暗示 FOV 到 BEV 的投影关系在预训练阶段已经学得足够好，后续只需调整检测头即可。

3.2 差异化融合深度

不同任务对 LiDAR 的依赖程度截然不同：

  • Occupancy 任务：以相机为主导，LiDAR 辅助校正深度。只使用 LiDAR 的 s4 单尺度特征，取最后一帧，通过多膨胀卷积融合
  • LidarGod 任务：以 LiDAR 为主导，相机补充语义。使用 s2/s4/s8 全尺度特征，多帧时序对齐聚合，并经过完整的 BEV Backbone + FPN 处理
  • OD 任务：同样使用全尺度融合 BEV 特征，但在检测头内部还额外通过可变形交叉注意力直接查询原始相机 FPN 特征

这种差异化设计反映了一个重要的工程直觉：不同任务对几何精度和语义丰富度的权衡不同。Occupancy 关注的是空间占据状态，相机的语义分割能力更关键；而 3D 检测需要精确的深度和尺寸信息，LiDAR 的几何优势不可替代。

四、稀疏检测头：多源 Query 与 Transformer 解码

检测头的设计是整个系统最体现架构思考的部分。它采用了 DETR 风格的集合预测范式，但在 Query 初始化和解码过程中做了大量创新。

4.1 多源 Query 初始化

每一帧的检测 Query 来自四个来源，总计约 948 个：

  • 可学习 Query（100 个）：网格初始化的通用 Query，覆盖全空间的先验
  • Mono3D Query（220 个）：前视单目 3D 检测头产生的候选，每个相机独立检测后提供初始化位置和特征
  • LiDAR Query（128 个）：从 LiDAR BEV 特征经 RPN 生成，训练时施加 20% 的 Dropout 以及 FOV 区域的 Dropout，防止模型过度依赖 LiDAR
  • 历史 Query（500 个）：从时序记忆库中提取的历史帧检测结果

这种多源设计的核心思想是互补与冗余：可学习 Query 提供稳定的空间先验；Mono3D Query 利用相机的纹理优势在远距离初始化目标；LiDAR Query 在近距离提供精确的几何锚点；历史 Query 则赋予系统时序连续性。

LiDAR Query 的 Dropout 策略尤其值得注意 —— 这是为了让 Transformer Decoder 学会在 LiDAR 信号缺失时（如恶劣天气或传感器故障）退化到仅相机模式，提高系统的鲁棒性。

4.2 Sparse Transformer Decoder

Decoder 只有 2 层（相比 DETR 通常的 6 层），每层的操作序列经过精心编排：

Query → 自注意力 → LiDAR 可变形交叉注意力 → 相机多尺度可变形交叉注意力 → FFN → 时序融合

先 LiDAR 后相机的顺序并非偶然：LiDAR 的可变形注意力在 BEV 空间的三个尺度特征图上采样，先建立精确的空间定位；然后相机的多尺度可变形注意力在原始透视图特征上采样，补充纹理和语义信息。这种先定位后识别的流程，与人类的感知逻辑一致。

4.3 后处理的双轨设计

系统维护了两套并行的后处理流程：

  • BEV 分支：NMS-Free 的集合预测后处理，直接按置信度阈值输出
  • LiDAR 分支：传统的 Anchor-based RPN + NMS 后处理

这种双轨设计是工程上的务实选择 —— NMS-Free 方法在理论上更优雅，但 LiDAR RPN 作为经过大量验证的成熟方案，提供了可靠的基线保障。

五、时序架构：Recurrent Temporal 的 4D 感知

时序建模是系统从"逐帧检测"升级为"4D 感知"的关键。

5.1 记忆库机制

系统维护一个记忆库，缓存过去多帧的检测结果。推理时采用 Recurrent 状态机模式：每帧处理完成后将 Top-500 个检测 Query（包含位置编码和特征嵌入）存入缓存，下一帧从缓存中取出 5 个关键帧（帧间隔 [-12, -9, -6, -3, -1]）的历史 Query。

关键帧的选取策略采用非均匀间隔 —— 近帧密集、远帧稀疏。这符合运动估计的信息论直觉：近帧提供高时间分辨率的运动细节，远帧提供长时间跨度的运动趋势。

5.2 时间对齐

历史 Query 的坐标需要从历史帧的自车坐标系变换到当前帧：

当前帧坐标 = global2cur × his2global × 历史帧坐标

这是一个纯 ego-motion 对齐，不做速度补偿。理由是：速度补偿本身就是检测头需要预测的目标之一，如果在输入端就做了补偿，会形成循环依赖。让模型自行学习从未补偿的历史轨迹中推断速度，是更干净的设计。

5.3 时序融合层

在 Transformer Decoder 的每一层末尾，新增了一个时序交叉注意力层：当前帧 Query 对历史记忆 Query 进行注意力查询。这使得模型能够显式地关联当前观测与历史轨迹，输出的速度和运动状态预测子头直接受益于这种时序信息。

六、训练策略的工程智慧

6.1 渐进式冻结

系统采用分阶段训练策略：在微调阶段，冻结视觉 Backbone、LiDAR Backbone、FPN、FOV2BEV 投影和 LiDAR 融合模块，只训练检测头、记忆库和时序相关模块。

这种策略的思考是：底层特征提取的泛化能力远强于上层任务头。预训练阶段在大规模数据上学到的特征表示足够通用，微调时只需调整任务特定的决策层。这大幅减少了训练参数量和计算资源需求。

6.2 多任务多步训练

多任务训练采用"逐任务前向"模式：每个优化步内，OD、Occupancy、LidarGod 三个任务依次前向传播、计算 Loss、反向传播，最后一次性更新参数。

这种设计相比"批内混合"有三个显著优势：

  • 梯度隔离：每个任务的梯度独立计算，避免不同任务的特征混叠
  • 显存峰值低：每次只保留单任务的中间激活值（BEV 特征图体积庞大）
  • 灵活冻结：不同任务可以冻结不同的共享模块，逐任务模式使控制更精确

6.3 训练时的时序模拟

训练时使用 3 帧时序输入（帧间隔 [-4, -2, 0]），并通过模拟序列采样来弥合"训练时看到完整序列"与"推理时只能逐帧递推"之间的差距。

七、关键设计洞见总结

回顾整个系统，几个贯穿始终的设计原则值得提炼：

1. 表示空间的精心选择

系统巧妙地利用了三种互补的空间表示：

  • Range View（距离图）：利用 LiDAR 扫描的自然结构，用 2D 卷积高效提取点级特征
  • Voxel Space（体素空间）：稀疏 3D 卷积提取几何结构特征
  • BEV Space（鸟瞰图空间）：统一融合多模态信息，便于下游任务消费

每种表示各有优劣，系统通过 Range View → Voxel → BEV 的渐进转化，在每个阶段使用最适合的计算范式。

2. 稀疏性的极致利用

从点云到体素到 BEV，系统始终在"稀疏"和"密集"之间做精确的切换：稀疏的地方（体素 3D 卷积）不浪费计算在空体素上；密集的地方（BEV 特征图）用于需要规整输入的融合和检测；检测头本身也是稀疏的 —— 只维护约 1000 个 Query，而非在整个 BEV 网格上做密集预测。

3. 可微性的全链路保障

从 Range View U-Net 到可微体素化，再到稀疏卷积和 Transformer，整条管线保持完整的梯度流。这不是自然而然的 —— 传统体素化的 CUDA 算子会切断梯度，需要有意识地用确定性累加操作来替代。

4. 退化鲁棒性的主动设计

LiDAR Query Dropout、多源 Query 冗余、以及独立的 LiDAR RPN 基线分支，都是为了让系统在部分传感器失效时仍能维持合理的检测能力。这种"为失败场景设计"的思维，是量产级系统与学术原型的根本区别。

5. 时序信息的层次化利用

系统在不同层次利用时序信息：特征级（BEV 特征的多帧时序对齐和融合）、Query 级（历史检测 Query 作为当前帧的初始化输入）、预测级（速度和运动状态的显式建模）。这种多层次设计确保时序信息在需要的地方都能被有效利用，而不是仅靠某个单一机制。

这套多模态稀疏 3D 检测系统展现了量产级自动驾驶感知架构的复杂性和精妙之处。它不是某篇论文的简单复现，而是无数工程决策的有机组合 —— 每一个看似微小的选择（体素化的梯度支持、距离图的非均匀采样、多源 Query 的 Dropout 策略）背后，都有深刻的技术考量和实践验证。

理解这样一个系统，不仅需要掌握单个技术点，更需要理解它们如何被系统性地组装在一起，以及为什么在面对无数可能的设计选择时，最终选择了当前的方案。这种从全局到局部、从原理到工程的思维方式，才是自动驾驶感知系统开发的核心竞争力。
