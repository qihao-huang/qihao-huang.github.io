多模态稀疏 3D 检测：工程哲学与架构设计

2026-04-10 · 返回首页

Made with Cursor: 把一套多模态 3D 检测系统做进工程里，团队通常会先遇到哪些问题，又为什么会一步步长成现在这样的样子。

一、先把大框架搭起来

一开始最重要的不是选哪个 fancy 模块，而是先把主流程想清楚：数据从哪里进来，中间在哪一层汇合，哪些部分要共享，哪些部分必须分开做。

传感器 → 单模态编码 → BEV 对齐与融合 → 稀疏集合式检测 → 时序状态更新

比较常见的做法是，前面先让各个传感器各自提特征，后面再往同一个 BEV 空间里对齐，最后交给检测头和时序模块。这样拆的好处很直接：主干部分可以复用，不同任务也能在后面各自演化，不用每次改一点东西就把整套系统推倒重来。

相机这边也一样。前视、侧视、后视的视角和职责差别很大，强行全塞进一个完全统一的分支，最后常常是谁都照顾不好。所以工程上更常见的是按相机位置分组处理，先承认输入本来就不一样，再在后面做融合。

二、LiDAR 这条线，核心是在“乱”里找秩序

点云最大的问题不是信息少，而是太乱了。它不是规则网格，远近疏密还不一样；但大多数网络又更擅长处理规整输入。所以工程上通常不会一上来就把点云硬塞进某一种表示里，而是分几步走，让每一步只解决一个问题。

第一步：先把点云变成容易处理的样子

因为 LiDAR 本来就是一圈一圈扫出来的，所以很多系统会先把点投成一张距离图。这样做不是为了好看，而是为了先借用成熟的 2D 卷积，把“附近有什么结构”这件事先看清楚。至于哪里分得细、哪里分得粗，本质上就是一句话：重要的地方多花点算力，不重要的地方别平均用力。

第二步：在图上聚一聚，再回到点上

点投到距离图之后，很多点会落在同一个位置上。这时可以先把它们聚在一起算特征，算完再分回每个点。你可以把它理解成：先在图像这层把局部上下文看明白，再把这份上下文还给原始点云。这样做比直接上来切体素更稳，因为很多大尺度关系在图上更容易看出来。

第三步：再进体素，但别把训练链路断掉

后面要做 3D 几何建模，还是得回到体素这类表示。但如果体素化这一步把梯度断了，那前面距离图那条支路很可能就学不动了。所以工程上会很在意这一段是不是可训练、可回传。

真正到了体素阶段，另一个现实问题就来了：空格子太多。如果还像 dense 卷积那样全部去算，成本太高，所以才会用稀疏卷积，只算有内容的地方。除此之外，还有两类很朴素但很重要的问题：结果能不能稳定复现，坐标和索引会不会在大网格下悄悄出错。很多系统最后不是败在大思路，而是败在这些细节不稳定。

第四步：把几何特征整理成 BEV，方便后面统一消费

最后把 LiDAR 特征整理到 BEV 上，本质上是为了让后面的检测、融合、时序都在同一个坐标系里说话。这里没有一套放之四海而皆准的配方，只有很现实的权衡：近处要不要更细，远处要不要保更大范围，高度信息要留多少，前向区域要不要裁掉一部分换算力。说到底，就是产品目标、算力预算和误检漏检容忍度在互相拉扯。

三、融合不是越早越好，而是看任务需要什么

很多人一说多模态，第一反应就是“把视觉和 LiDAR 拼起来”。但工程里真正要问的是：这个任务到底更缺几何，还是更缺语义。比如有些任务更依赖空间形状，那 LiDAR 就该更靠前；有些任务更依赖语义分类，那视觉就要多承担一些。融合做多深、做多早，不是默认选项，而是看任务目标。

还有一个很现实的问题：有些几何投影模块前期已经训得比较稳了，后面微调时继续乱动，反而容易把系统搞乱。所以冻结某些模块，通常不是偷懒，而是团队已经判断当前瓶颈不在那里。

四、检测头真正要解决的是“候选从哪来，最后信谁”

不管外面包装成什么范式，检测头最后都在解决几个老问题：候选目标从哪里来、不同来源冲突时听谁的、某个传感器掉链子时能不能顶住。

为什么会走到 sparse query 这条路

传统做法更像“把整张 BEV 地图都扫一遍”，每个位置都试着预测目标；这当然直接，但也很重，因为大部分位置其实什么都没有。sparse query 的思路正好相反：别把注意力平均撒满全图，而是只维护少量值得重点看的候选，让模型围着这些候选去做匹配、更新和筛选。

这套范式在工程上有几个很实际的好处。第一，算力更集中，不用在大片空区域上反复做无效计算。第二，结构更容易接多源信息，因为每个 query 都可以看成一个“待确认目标”，它可以同时吸收 LiDAR、视觉和历史状态的信息。第三，时序也更自然，因为跨帧传递的不是整张大特征图，而是一组已经比较像“目标状态”的表示。

当然，sparse query 也不是白拿来的。你得回答这些 query 一开始从哪来，后面怎么更新，什么时候该删，什么时候该补。所以它不是简单地把 dense 检测头换个名字，而是把问题从“全图打点”改成了“维护一小群候选目标”。

候选最好别只靠一路

更稳的做法是让候选同时来自几路：有的来自可学习先验，有的来自视觉分支，有的来自 LiDAR，还有的来自历史状态。这样做的原因很简单，真实路况里没有哪个来源永远可靠。多放几路，不是为了显得复杂，而是为了让系统在某一路变差时还有兜底。

顺序上通常先定位，再补语义

很多系统会先用 LiDAR 把空间位置卡住，再让视觉补类别和细节。原因不复杂：几何错了，后面语义再强也救不回来。至于解码器做多深，这通常不是“论文里几层最好”，而是延迟、显存和收益一起看的工程决定。

后处理可以两条腿走路

工程上常见的做法是，新范式继续往前推，传统路线先别急着扔。前者帮你探索上限，后者帮你兜住下限。系统能回滚、能对比、能解释，很多时候比“结构看起来更统一”更重要。

五、时序不是多喂几帧，而是管理“上一帧留下了什么”

很多系统做时序，问题不在于喂多少帧，而在于上一帧到底留下什么给下一帧。是留目标状态，还是留中间特征；读最近几帧，还是拉更长历史；只做自车坐标对齐，还是额外做运动补偿。这些选择决定了系统到底是在“看连续视频”，还是只是“把几帧拼一起”。

当前结果 = 本帧观测 + 对齐后的历史信息

六、训练阶段，核心是别把系统一起搅乱

真正落地时，训练不是“所有模块一起放开，端到端猛训”这么简单。很多时候底层特征已经够通用了，真正该动的是后面的任务头和时序部分。多任务一起训时，也要小心它们互相打架，所以常会拆开前向、统一更新，尽量把问题隔离开。还有一个常见坑：训练时看的是一套时序分布，推理时跑的是另一套递推逻辑，如果中间不补这层差距，离线指标和真实效果经常对不上。

七、最后总结几句更接地气的话

  • 表示没有银弹：距离图、体素、BEV 各有各的用处，关键是接得顺不顺。
  • 算力别浪费在空气上：能稀疏就稀疏，能裁剪就裁剪。
  • 训练链路别断：某一步不能回传，前面很可能就白做。
  • 融合看任务，不看流行：任务要什么，就让谁多说话。
  • 冗余不是浪费，是保险：多一路候选、多一条后处理，很多时候能救命。
  • 时序要管状态，不只是堆帧：系统记住什么，比系统看了几帧更重要。

如果把这篇文章压成一句话，那就是：这类系统不是靠某一个神模块赢的，而是靠很多看起来不那么“炫”的工程决定一点点拼出来的。哪些地方先做规整，哪些地方保留冗余，哪些地方省算力，哪些地方换稳定性，这些选择最后共同决定了系统好不好用。

所以比起背参数，我更关心的是：如果今天换一套传感器，或者目标从论文指标变成真实量产，这条链路里哪些地方最先要改。把这个想清楚，比记住几个模块名字更有用。
